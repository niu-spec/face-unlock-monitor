**B组反馈：当前后端 Native 崩溃情况**
当前生产环境发现 **Gunicorn Worker 原生内存崩溃**，问题尚未最终解决。
**1\. 异常表现**
后端日志出现：

```
double free or corruption (out)
free(): corrupted unsorted chunks
Gunicorn worker exited with code 134
```

Worker 崩溃后 Gunicorn 会自动拉起新 Worker，所以：

```
/api/video/status
/api/video/presence
```

仍可能继续返回 200，但这不代表服务稳定。
实际表现为：
stream\_id=1 起初可以正常读取；
运行一段时间后 last\_frame\_at 停止刷新；
stream\_live 从 true 变为 false；
stream\_id=2 相对正常；
Worker 被重新拉起后可能暂时恢复。
**2\. 已排除的问题**
目前已确认不是以下原因：
不是 OBS 推流地址错误；
不是 MediaMTX 路径错误；
不是 RTSP 端口故障；
不是 Nginx 反代故障；
不是数据库迁移问题；
不是 ffmpeg 录制问题。
已验证：

```
rtsp://127.0.0.1:8554/stream/1
OpenCV opened: True
frame: (360, 640, 3)
```

MediaMTX 日志也能看到：

```
stream/1 is available and online
is publishing to path 'stream/1'
is reading from path 'stream/1'
```

之前的 RTSP 404 是 OBS 尚未推流时产生的历史日志，当前没有持续 RTSP 404。
**3\. 初步判断**
code 134、double free 和 corrupted unsorted chunks 属于 C/C\+\+ 原生层内存错误，不是普通 Python 异常。
当前怀疑方向包括：
同一个 cv2\.VideoCapture 被多个线程同时操作；
一个线程执行 read\(\) 时，另一个线程执行 release\(\)；
RTSP 重连时旧读取线程尚未退出，新 capture 已经创建；
OpenCV frame 底层缓冲区被多个异步任务共享，没有及时 frame\.copy\(\)；
dlib、face\_recognition 或其他模型对象被多线程并发调用；
stream\_id=1 和 living\_room 被重复注册成两个 Worker，但实际读取同一路流；
视频 Worker、音频检测和告警短视频逻辑之间存在生命周期冲突。
目前只是初步判断，尚无 core dump 能证明具体是哪个原生库。
