## 一、项目概述

### 1.1 建设目标

本项目面向**居家智能安防**场景，基于家庭摄像头实时视频流，构建一套可落地的监测系统。系统实现家庭成员人脸注册与识别、陌生人告警、危险区域入侵管控，以及着火、人员摔倒等异常事件检测，并通过统一告警中心完成事件展示、处置与回放。可选能力方面，系统进一步集成人脸反欺骗检测、异常声学识别、钉钉逐级上报与 AI 监控日报等模块。

### 1.2 技术路线


| 层次        | 选型                                                               |
| --------- | ---------------------------------------------------------------- |
| 前端        | Vue3 + Vite + Element Plus                                       |
| 业务与 AI 后端 | Django 4.2 + DRF + SimpleJWT（单体集成）                               |
| 视觉算法      | dlib / face_recognition、YOLOv8n（HOG 降级）、OpenCV                   |
| 音频算法      | PANNs（异常声学，可选）                                                   |
| 流媒体       | OBS → RTMP → MediaMTX → RTSP；浏览器主预览 WebRTC + Canvas Overlay      |
| 数据存储      | MySQL                                                            |
| 工程化       | GitHub、OpenSpec、Swagger（drf-yasg）、GitHub Actions CI + Jenkins CD |


整体采用「采集端推流 → 流媒体转发 → Django 帧处理链 → 业务入库 / Overlay 展示」的数据路径，业务 API 与 AI 模块同进程部署，降低联调与运维复杂度。

## 三、关键技术与重难点

### 3.1 视频预览与 AI 标注解耦

早期方案以 MJPEG 烧录标注为主，延迟偏高、前端扩展受限。现采用 **WebRTC 展示原始画面 + 前端 Canvas 轮询 overlay API** 叠加人脸框 / 告警框，既降低观感延迟，又便于独立迭代 UI。

### 3.2 多人协作下的帧处理链

`process_frame()` 串联人脸识别、区域违规、异常检测与告警写库。通过 OpenSpec 约定接口与告警类型命名，以及家庭 / 摄像头数据隔离，减少模块互相覆盖与跨组串扰。

### 3.3 检测精度与实时性平衡

行人检测优先 YOLOv8n，环境受限时降级 HOG；火情与摔倒采用可调阈值 + 冷却，降低误报刷屏；音频模块在无音轨或依赖缺失时降级跳过，保证主视频链路可用。

### 3.4 生产部署与 CI/CD

云环境面对依赖安装、推流连通、Worker 线程安全、Camera 绑定等问题，通过部署脚本、运维清单与 Jenkins / GitHub Actions 逐步固化发布路径，支撑全员可复现部署。

## 六、遇到的困难、不足与展望

### 6.1 开发与联调中遇到的主要困难

**（1）视频预览延迟与架构切换**  
立项初期采用后端 MJPEG 烧录标注推前端，联调中发现延迟高、前端难以独立叠加多类框。团队随后将主预览改为 **MediaMTX WebRTC + 前端 Canvas 轮询 overlay**，MJPEG 降为备用。切换过程中出现过坐标缩放不一致、分路 presence 串流等问题，经 A/B/C 联合对齐 `stream_id` 与画布尺寸后解决。

**（2）云上推流「无画面」排查成本高**  
现象多为监控页空白或 Worker 无帧，根因分散在 OBS 推流地址、安全组端口、MediaMTX 状态、Camera 未绑定家庭、后端未 pull 最新代码等环节。通过固化「推流 → 中继 → 拉流 → presence」分层排查清单与 Camera 补配置指引，缩短了中后期排障时间。

**（3）本地 / 生产 AI 依赖安装困难**  
dlib、YOLO、PyTorch 等在 Windows 开发机与 Linux 生产机上安装差异大；音频相关一度引入过重的 `torchaudio` 依赖导致部署失败。处理方式：精简 requirements、提供 conda / pip 脚本与「依赖修复重部署」文档，生产侧先 pull 再 `pip install --no-cache-dir` 后重启。

**（4）视频 Worker 稳定性**  
长时间拉流时出现过原生库 / 多线程不安全导致的进程异常。B 组对 `CameraWorker` 做采集与推理分线程及线程安全修复，并配合重启策略与日志定位，保障结题演示可持续推流。

**（5）多模块接口约定不一致**  
人脸角色、禁区规则、告警类型命名在并行开发初期曾出现文档与代码漂移（如早期 MJPEG/ZONE 命名与现行 INTRUSION 等不一致）。依靠 OpenSpec 固化行为规格、统一 `create_alert` 写库入口，并以告警中心 / Swagger 作为验收对照，减少了口头对接误差。

**（6）检测误报与场景标定**  
室内光线、摄像头俯仰角对火情色阈、摔倒高宽比影响显著；禁区像素坐标与检测帧分辨率不一致会导致「画了区却不报」。通过可配置阈值、告警冷却，以及 ZoneEditor 与检测帧的坐标缩放修复缓解；演示前需按实际机位再标定一次。

**（7）部署流水线与路径配置**  
Jenkins / rsync、Nginx 前端静态目录指向、管理命令未注册等问题在云部署中多次出现，表现为「CI 绿了但页面仍是旧版」。通过部署错误修复文档、统一生产目录策略和人工确认门禁逐步收敛。

**（8）文档与代码同步滞后**  
架构从 MJPEG 主路径升级为 WebRTC 后，中期文档若未及时改写，答辩材料会与系统不符。F 组以变更清单驱动 v3.0 修订，要求各组按时回传截图与接口变更，仍是结题阶段需要持续盯紧的事项。

### 6.2 仍存在的不足

1. **智慧交通沙盘未做**：本项目聚焦居家场景，未对接沙盘设备做隧道 / 停车场等扩展验证。
2. **可选能力对环境敏感**：异常声学依赖音轨与模型权重，AI 日报依赖可选 LLM Key，钉钉依赖真实 Webhook；环境未配齐时易出现「功能在代码中、演示不完整」。
3. **测试覆盖仍可加强**：核心链路已有单测与云上 E2E 演示，但长稳压测、多路并发与更多攻击样本集的系统评测不足。

### 6.3 展望

短期完成结题提交与验收演示固化；中期可考虑引入更强的活体模型、完善监控大盘与告警分析，并将 CI/CD 做到更少人工干预的一键发布。若条件允许，再用沙盘或更多真实家居机位做跨场景回归，提升算法泛化与产品完整度。

## 七、总结

经过约两周的需求分析、设计、实现与云上联调，本组完成了居家智能摄像头监控系统的核心闭环：**识人 → 划区 → 异常检测 → 告警处置与回放**，并在工程规范（Git、OpenSpec、Swagger、CI/CD）与可选加分项上取得实质性进展。团队在分工明确的前提下通过规格与接口协作，在有限周期内交付了可演示、可部署的系统。
